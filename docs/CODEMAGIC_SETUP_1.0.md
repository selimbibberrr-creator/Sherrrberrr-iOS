# Codemagic setup — SherrrBerrr iOS 1.0

## A. First compile — no Apple signing required

1. Add/connect GitHub repository `selimbibberrr-creator/Sherrrberrr-iOS` in Codemagic.
2. Select branch `main` and use **Check for configuration file**. Codemagic must detect root `codemagic.yaml`.
3. Start workflow **SherrrBerrr 1.0 - iOS Simulator Compile**.
4. A successful run materializes the validated 1.0 source tree, builds both `GabberShermanKick_AUv3` and `GabberShermanKick_Standalone`, exposes the simulator artifact ZIP, and publishes `artifacts/metadata/bundle_ids.txt`.
5. Open `bundle_ids.txt`. It contains the exact container bundle ID and JUCE-generated AUv3 child bundle ID. Register both identifiers in Apple Developer Portal if they do not already exist before attempting signed builds.

Do not troubleshoot signing before this workflow is green; it isolates source/Xcode/JUCE errors from Apple-account errors.

## B. Apple credentials for an installable iPad IPA

A paid Apple Developer Program membership is required for Codemagic-managed signing.

In App Store Connect, create an API key with sufficient Developer Portal/App Manager rights. In Codemagic create an environment variable group named exactly `appstore_credentials` with secret values:

- `APP_STORE_CONNECT_PRIVATE_KEY` — full contents of the downloaded `.p8` key;
- `APP_STORE_CONNECT_KEY_IDENTIFIER` — Key ID;
- `APP_STORE_CONNECT_ISSUER_ID` — Issuer ID;
- `CERTIFICATE_PRIVATE_KEY` — PEM private key used by Codemagic to match or create the Apple signing certificate.

Register the iPad UDID in the Apple Developer account before the Development workflow if it is not already registered. Also ensure both bundle identifiers from the smoke-build artifact are registered; AUv3 extensions require a separate provisioning profile.

Run **SherrrBerrr 1.0 - Signed iPad Development IPA**. The workflow:

1. materializes and validates the 1.0 source contract;
2. pins/fetches JUCE 8.0.10;
3. generates the Xcode iOS project;
4. discovers the actual Standalone + AUv3 bundle IDs generated by JUCE;
5. creates/fetches Development profiles for both IDs;
6. signs and exports the IPA;
7. unpacks the IPA and fails if no embedded `.appex` is present.

The resulting `.ipa` is the first physical-iPad test build.

## C. TestFlight internal

Create the app record in App Store Connect for bundle ID `com.selimbibberrr.sherrrberrr` before running this workflow.

Run **SherrrBerrr 1.0 - TestFlight Internal Upload**. It uses App Store distribution profiles and marks the export for TestFlight Internal Testing Only. The build is uploaded automatically after a successful IPA/AUv3 verification. `submit_to_testflight` remains false because internal-only builds do not require external TestFlight beta review; assign the processed build to your internal tester group in App Store Connect.

## Fixed build inputs

- Xcode: `26.4`
- JUCE: `8.0.10`
- Minimum iOS/iPadOS: `15.0`
- User-facing version: `1.0.0`
- CI build version: `1.0.$CM_BUILD_NUMBER`

These are intentionally pinned so a later toolchain update cannot silently change a 1.0 build.
